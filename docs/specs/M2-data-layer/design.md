# Design Document: M2 — Data Layer

## Overview

M2 establishes the persistence layer for Stock Plan Manager's medallion architecture. The schema is structured around financial instrument lifecycles rather than a single polymorphic table — RSU/ESOP/ESPP each have distinct lifecycle mechanics handled by purpose-built tables (origins, tranches, exercises) that flow into a shared sales + allocation system. Seven tables total: 2 Bronze (ingestions, bronze_raw), 5 Silver (origins, tranches, exercises, sales, sale_allocations).

### Key Design Principles

1. **Lifecycle-driven schema**: Tables follow the financial lifecycle, not arbitrary abstraction. Different instruments get different lifecycle paths.
2. **Origins + Tranches + Exercises model**: Corporate events (grants, vests) are separate from user-triggered events (exercises, sells).
3. **Shared sales system**: All plan types sell through the same sales + sale_allocations tables.
4. **metadata_json for type-specific fields**: Keeps shared columns clean; type-specific details in JSON.
5. **SafeDecimal everywhere**: Financial amounts stored as TEXT, loaded as `%Decimal{}`.
6. **FK enforcement with restrict**: DB-level integrity. Deletes must be explicit and ordered.
7. **Schemas are dumb**: Zero business logic — only fields and changeset validation.

### Architecture

```
stock_plan_ingestions (file tracking)
     |
     |── stock_plan_bronze_raw (raw audit)
     |
     |── stock_plan_origins (RSU grant / ESOP grant / ESPP enrollment)
     |        |
     |        |── stock_plan_tranches (RSU/ESOP vest schedule)
     |        |        |
     |        |        |── stock_plan_exercises (ESOP only)
     |        |
     |── stock_plan_sales (sell executions, all types)
              |
              |── stock_plan_sale_allocations (lot linkage)
```

### Lifecycle Flows

```
RSU:   origin (grant)      → tranche (vest)      → sale_allocation (TRANCHE) ← sale
ESPP:  origin (enrollment) → tranche (purchase)   → sale_allocation (TRANCHE) ← sale
ESOP:  origin (grant)      → tranche (vest)       → exercise → sale_allocation (EXERCISE) ← sale
```

All plan types flow through tranches. ESOP additionally requires exercise to convert rights into shares.

## Components and Interfaces

### 1. SafeDecimal (`lib/stock_plan/types/safe_decimal.ex`)

Unchanged from prior implementation. Bridges `%Decimal{}` ↔ SQLite TEXT.

### 2. ID Generator (`lib/stock_plan/id.ex`)

Unchanged. `:crypto.strong_rand_bytes(8) |> Base.encode16(case: :lower)`.

### 3. Origin Schema (`lib/stock_plan/schema/origin.ex`)

```elixir
defmodule StockPlan.Schema.Origin do
  use Ecto.Schema
  import Ecto.Changeset

  @primary_key {:id, :string, autogenerate: false}
  schema "stock_plan_origins" do
    field :ingestion_id, :string
    field :account_id, :string
    field :symbol, :string
    field :plan_type, :string
    field :grant_number, :string
    field :origin_date, :date
    field :total_quantity, StockPlan.Types.SafeDecimal
    field :origin_fmv, StockPlan.Types.SafeDecimal
    field :origin_fx_rate, StockPlan.Types.SafeDecimal
    field :currency, :string
    field :status, :string
    field :metadata_json, :string

    timestamps(type: :utc_datetime_usec)
  end

  @required ~w(id ingestion_id account_id symbol plan_type origin_date currency)a
  @optional ~w(grant_number total_quantity origin_fmv origin_fx_rate status metadata_json)a
  # total_quantity: required for RSU/ESOP (enforced in M5 Silver builder), nil for ESPP
  # grant_number: broker-assigned for RSU/ESOP, computed hash for ESPP

  def changeset(struct, attrs) do
    struct
    |> cast(attrs, @required ++ @optional)
    |> validate_required(@required)
    |> validate_inclusion(:plan_type, ~w(RSU ESPP ESOP))
    |> validate_inclusion(:currency, ~w(USD))
    |> unique_constraint([:ingestion_id, :grant_number])
  end
end
```

**ESPP grant_number:** Computed as first 16 chars of `SHA256("ESPP:{symbol}:{origin_date}")`. Provides stable dedup key since ESPP has no broker-assigned grant number. Generated by Silver builder (M5) during ingestion.

**Unique constraint:** `(ingestion_id, grant_number)` — prevents duplicate grant imports. ESPP has nullable grant_number so this only applies to RSU/ESOP.

### 4. Tranche Schema (`lib/stock_plan/schema/tranche.ex`)

```elixir
defmodule StockPlan.Schema.Tranche do
  use Ecto.Schema
  import Ecto.Changeset

  @primary_key {:id, :string, autogenerate: false}
  schema "stock_plan_tranches" do
    field :origin_id, :string
    field :ingestion_id, :string
    field :vest_date, :date
    field :vest_quantity, StockPlan.Types.SafeDecimal
    field :vest_fmv, StockPlan.Types.SafeDecimal
    field :vest_fx_rate, StockPlan.Types.SafeDecimal
    field :tax_withheld_qty, StockPlan.Types.SafeDecimal
    field :net_quantity, StockPlan.Types.SafeDecimal
    field :status, :string
    field :metadata_json, :string

    timestamps(type: :utc_datetime_usec)
  end

  @required ~w(id origin_id ingestion_id vest_date vest_quantity status)a
  @optional ~w(vest_fmv vest_fx_rate tax_withheld_qty net_quantity metadata_json)a
  @statuses ~w(UNVESTED VESTED FORFEITED CANCELLED EXPIRED)
  # Quantity invariants (enforced in Silver builder, not schema):
  # tax_withheld_qty <= vest_quantity
  # net_quantity = vest_quantity - tax_withheld_qty
  # available_qty = net_quantity - sum(sale_allocations.quantity) [derived at query time]

  def changeset(struct, attrs) do
    struct
    |> cast(attrs, @required ++ @optional)
    |> validate_required(@required)
    |> validate_inclusion(:status, @statuses)
  end
end
```

**No unique constraint on tranches.** Split vests, corrections, or multiple ESPP purchases on the same date are possible. Dedup handled by Silver builder (M5) import logic, not DB constraint.

## Data Models

### Foreign Key Graph

```
ingestions ← bronze_raw
ingestions ← origins ← tranches ← exercises
ingestions ← sales ← sale_allocations
```

`sale_allocations` uses real DB FKs: `tranche_id` (NOT NULL) and `exercise_id` (nullable). No polymorphism.

### metadata_json Structures

```jsonc
// Origin: RSU
{}

// Origin: ESPP
{"lock_in_price": "150.00", "buy_price": "127.50", "discount_percent": "15", "buy_fmv": "160.00"}

// Origin: ESOP
{"strike_price": "72.36", "option_type": "NQ", "expiry_date": "2030-03-15"}

// Tranche metadata (RSU vest)
{"tax_description": "Federal + State", "effective_tax_rate": "0.37", "taxable_gain": "11250.00"}

// Exercise metadata
{"tax_description": "AMT adjustment", "bargain_element": "5000.00"}

// Sale metadata
{"broker_fee": "4.95", "trade_confirmation_ref": "TC-2025-001234"}
```

## Correctness Properties

### Property 1: SafeDecimal Round-Trip
*For any* valid Decimal, dump → load returns original value.

### Property 2: ID Format
*For any* `StockPlan.ID.generate/0`, result matches `^[0-9a-f]{16}$`.

### Property 3: FK Integrity
*For any* tranche, its `origin_id` must reference an existing origin. DB enforces via FK constraint.

### Property 4: Tranche Status Semantics
*For any* tranche with status UNVESTED, `vest_fmv` must be nil. *For any* tranche with status VESTED, `vest_fmv` should be populated.

### Property 5: Sale Allocation Quantity Sum
*For any* sale, the sum of `sale_allocations.quantity` should equal `sales.total_quantity`.

### Property 6: Quantity Invariants (enforced in M5, not DB)
- *For any* tranche: `tax_withheld_qty` ≤ `vest_quantity` and `net_quantity` = `vest_quantity` - `tax_withheld_qty`
- *For any* exercise: `tax_withheld_qty` ≤ `exercise_quantity` and `net_quantity` = `exercise_quantity` - `tax_withheld_qty`
- *For any* tranche (ESOP): sum of `exercise_quantity` for all exercises ≤ tranche `net_quantity` (can't exercise more than vested)
- *For any* allocation: sum of `allocations.quantity` by tranche ≤ `tranche.net_quantity` (RSU/ESPP)
- *For any* allocation: sum of `allocations.quantity` by exercise_id ≤ `exercise.net_quantity` (ESOP)

### Property 7: Available Quantity (Derived, NEVER stored)
*For any* lot: available = `net_quantity` - sum of linked `sale_allocations.quantity`. Computed at query time.

## metadata_json Governance

Allowed keys per table (enforced by Silver builder, not schema):

```
origins:
  RSU:  {}  (no type-specific metadata needed)
  ESPP: {discount_percent, qualified_plan, contribution_source}
  ESOP: {strike_price, option_type, expiry_date}

tranches:
  RSU:  {tax_description, effective_tax_rate, taxable_gain}
  ESPP: {buy_price}  ← discounted purchase price = cost basis
  ESOP: {}  (vest just unlocks right, no financial data)

exercises:
  ESOP: {tax_description, bargain_element}

sales:
  {broker_fee, trade_confirmation_ref}
```

Rules: No dates (use typed columns). No prices you'll query often (those are columns). metadata_json is for supplementary/one-off fields only.

## Error Handling

| Error Condition | Handling Strategy | Caller Impact |
|---|---|---|
| SafeDecimal cast invalid | `:error` returned | Changeset validation error |
| FK violation on insert | `Ecto.ConstraintError` raised | SQLite doesn't expose FK names — catch at Repo level |
| Unique constraint violation (bronze row_hash) | `{:error, changeset}` via `unique_constraint/3` | Context handles |
| Invalid exercise_id for non-ESOP | Application validation in M5+ | Schema allows it; builder enforces |

## Testing Strategy

| Test Type | Coverage | Key Scenarios |
|---|---|---|
| Unit | SafeDecimal, ID | Cast/dump/load, format, uniqueness |
| Unit | Changesets | Required fields, inclusion validation, optional fields |
| Integration | Repo round-trip | SafeDecimal, :date, timestamps, FK enforcement |
| Integration | Lifecycle | Origin → Tranche → Exercise → Sale → Allocation chain |

## Implementation Notes

- Existing M2 migrations and schemas must be deleted before creating new ones (no data exists).
- `after_connect` FK pragma already configured from prior M2 work — keep it.
- SQLite doesn't expose FK constraint names — `foreign_key_constraint/3` in Ecto won't work. Test FK enforcement via `assert_raise Ecto.ConstraintError`.
- `sale_allocations` has real FK constraints on both `tranche_id` and `exercise_id`. No polymorphism.
- `metadata_json` is a plain `:string` field. Parsing/validation happens in Silver builder (M5), not in schema.
- `net_quantity` on tranches and exercises is computed by the Silver builder, not by the schema.
