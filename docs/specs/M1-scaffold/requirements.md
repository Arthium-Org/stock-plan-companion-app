# Requirements Document: M1 — Project Scaffold

## Introduction

The Project Scaffold module bootstraps the Stock Plan Manager as a Phoenix 1.8 LiveView application with SQLite persistence, Tailwind v4 styling, and Bandit HTTP server on port 4002. This is the foundation module — every subsequent module (M2–M13) depends on a bootable, correctly configured app. The scaffold must match the directory structure and conventions defined in CLAUDE.md with zero unnecessary extras (no auth, no mailer, no dashboard).

## Glossary

- **Phoenix_App**: The Elixir/Phoenix web application generated by `mix phx.new`
- **Bandit**: The HTTP server adapter replacing the default Cowboy in Phoenix 1.8
- **ecto_sqlite3**: The Ecto adapter for SQLite3 databases
- **SQLite_DB**: The SQLite database file at `tmp/stock_plan_dev.db`
- **Tailwind_V4**: Tailwind CSS version 4, the styling framework used by the app
- **LiveView**: Phoenix LiveView, the server-rendered reactive UI framework
- **Root_Route**: The default `GET /` route that confirms the app is serving pages
- **Dev_Server**: The Phoenix development server started via `mix phx.server`

## Requirements

### Requirement 1: Phoenix App Generation

**User Story:** As a developer, I want a clean Phoenix 1.8 app generated with the correct options, so that I have a minimal foundation without unused features.

#### Acceptance Criteria

1. THE Phoenix_App SHALL be generated with `--database sqlite3` to use ecto_sqlite3
2. THE Phoenix_App SHALL be generated with `--no-mailer` to exclude email functionality
3. THE Phoenix_App SHALL be generated with `--no-dashboard` to exclude Phoenix Dashboard
4. THE Phoenix_App SHALL include LiveView support (default in Phoenix 1.8)
5. THE Phoenix_App SHALL NOT include any authentication scaffolding
6. THE Phoenix_App SHALL compile with zero warnings from application code. Warnings from external dependencies or Elixir version mismatches are excluded but SHALL be documented if present.

### Requirement 2: HTTP Server Configuration

**User Story:** As a developer, I want the app to run on Bandit at port 4002, so that it doesn't conflict with other local services and uses the modern HTTP server.

#### Acceptance Criteria

1. THE Phoenix_App SHALL use Bandit as the HTTP server adapter (not Cowboy). IF Phoenix 1.8 generates Bandit by default, THE Phoenix_App SHALL use the generated config without manual override.
2. WHEN the Dev_Server is started, THE Phoenix_App SHALL listen on port 4002
3. WHEN the Dev_Server is started, THE Phoenix_App SHALL be accessible at `http://localhost:4002`
4. IF the generated config already sets the Bandit adapter, THE Phoenix_App SHALL NOT add a redundant `adapter: Bandit.PhoenixAdapter` line. Only override if the generated default is not Bandit.

### Requirement 3: Database Configuration

**User Story:** As a developer, I want SQLite configured with the correct database path, so that the database is created in the right location and is gitignored.

#### Acceptance Criteria

1. THE Phoenix_App SHALL use ecto_sqlite3 as the Ecto database adapter
2. THE SQLite_DB SHALL be located at `tmp/stock_plan_dev.db` in development
3. WHEN `mix ecto.create` is run, THE SQLite_DB SHALL be created at the configured path
4. WHEN `mix ecto.migrate` is run with no pending migrations, THE command SHALL succeed with no errors
5. THE `tmp/` directory SHALL be excluded from version control via `.gitignore`

### Requirement 4: Tailwind v4 Styling

**User Story:** As a developer, I want Tailwind v4 configured and working, so that all UI modules (M9–M12) can use utility-first CSS from the start.

#### Acceptance Criteria

1. THE Phoenix_App SHALL use Tailwind CSS version 4
2. WHEN the Dev_Server is started, THE Tailwind_V4 build SHALL compile CSS without errors
3. WHEN a page is loaded in the browser, THE Tailwind_V4 utility classes SHALL be applied correctly
4. THE Tailwind_V4 configuration SHALL be in `assets/css/app.css`

### Requirement 5: Root Route and Landing Page

**User Story:** As a developer, I want a root route that renders a basic page, so that I can verify the full stack (Phoenix + LiveView + Tailwind + SQLite) is working end-to-end.

#### Acceptance Criteria

1. WHEN a user visits `GET /`, THE Phoenix_App SHALL render a page without errors
2. THE Root_Route SHALL be defined in `lib/stock_plan_web/router.ex`
3. THE root page SHALL display the application name "Stock Plan Manager"
4. THE root page SHALL use Tailwind_V4 utility classes for basic styling
5. THE root page SHALL be a LiveView (not a dead view), validating LiveView is wired up

### Requirement 6: Directory Structure

**User Story:** As a developer, I want the project directory structure to match the CLAUDE.md specification, so that all subsequent modules have a consistent home.

#### Acceptance Criteria

1. THE Phoenix_App SHALL have `lib/stock_plan/` for business logic modules
2. THE Phoenix_App SHALL have `lib/stock_plan_web/` for web layer modules
3. THE Phoenix_App SHALL have `lib/stock_plan_web/live/` for LiveView modules
4. THE Phoenix_App SHALL have `lib/stock_plan_web/controllers/` for controllers
5. THE Phoenix_App SHALL have `lib/stock_plan_web/components/` for shared components
6. THE Phoenix_App SHALL have `priv/repo/migrations/` for database migrations
7. THE Phoenix_App SHALL have `config/` with `config.exs`, `dev.exs`, `test.exs`, and `runtime.exs`
8. THE Phoenix_App SHALL have `assets/css/app.css` and `assets/js/app.js`

### Requirement 7: Development Workflow

**User Story:** As a developer, I want all standard development commands to work, so that I can iterate quickly during implementation of subsequent modules.

#### Acceptance Criteria

1. WHEN `mix deps.get` is run, THE command SHALL install all dependencies without errors
2. WHEN `mix compile` is run, THE command SHALL compile with zero warnings from application code. No unresolved application warnings allowed.
3. WHEN `mix ecto.create` is run, THE command SHALL create the SQLite database
4. WHEN `mix ecto.migrate` is run, THE command SHALL apply pending migrations
5. WHEN `mix ecto.reset` is run, THE command SHALL drop, recreate, and migrate the database
6. WHEN `mix phx.server` is run, THE Dev_Server SHALL start and serve the Root_Route
7. WHEN `mix test` is run, THE test suite SHALL pass with zero failures

### Requirement 8: Application Naming

**User Story:** As a developer, I want consistent module naming throughout the app, so that all modules (M2–M13) follow the same namespace convention.

#### Acceptance Criteria

1. THE top-level application module SHALL be `StockPlan`
2. THE web module SHALL be `StockPlanWeb`
3. THE Ecto Repo module SHALL be `StockPlan.Repo`
4. THE Phoenix Endpoint module SHALL be `StockPlanWeb.Endpoint`
5. THE OTP application name SHALL be `:stock_plan`
