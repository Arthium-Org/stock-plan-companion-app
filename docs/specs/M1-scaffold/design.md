# Design Document: M1 — Project Scaffold

## Overview

M1 creates the foundational Phoenix 1.8 application for Stock Plan Manager. The scaffold produces a bootable app with SQLite persistence (ecto_sqlite3), Bandit HTTP server on port 4002, Tailwind v4 styling, and LiveView enabled. This module has no business logic — its sole purpose is to give M2–M13 a correctly configured platform to build on.

### Key Design Principles

1. **Minimal**: No features beyond what's needed to boot and verify the stack. No auth, no mailer, no dashboard.
2. **Convention-first**: Follow Phoenix 1.8 defaults wherever possible. Only override where CLAUDE.md requires it (port, DB path, server adapter).
3. **Verify everything**: Every configuration change is validated by a concrete check (server boots, page renders, CSS compiles, tests pass).

### Architecture

```
mix phx.new stock_plan --database sqlite3 --no-mailer --no-dashboard
     |
     v
┌─────────────────────────────────────────┐

```

## Components and Interfaces

### 1. Application Module (`lib/stock_plan/application.ex`)

**Responsibility**: Define the OTP supervision tree — starts Repo, Endpoint, PubSub, and Finch.

**Behavior**:
1. Standard Phoenix application module generated by `mix phx.new`
2. Starts `StockPlan.Repo` (SQLite connection)
3. Starts `StockPlanWeb.Endpoint` (Bandit on port 4002)
4. No custom children needed for M1

### 2. Repo Module (`lib/stock_plan/repo.ex`)

**Responsibility**: Ecto repository for SQLite database access.

```elixir
defmodule StockPlan.Repo do
  use Ecto.Repo,
    otp_app: :stock_plan,
    adapter: Ecto.Adapters.SQLite3
end
```

**Behavior**:
1. Uses `Ecto.Adapters.SQLite3` adapter
2. Database path configured per environment in config files
3. Dev database: `tmp/stock_plan_dev.db`
4. Test database: `tmp/stock_plan_test.db`

### 3. Endpoint (`lib/stock_plan_web/endpoint.ex`)

**Responsibility**: HTTP endpoint — serves static assets, WebSocket, and routes.

**Behavior**:
1. Standard Phoenix endpoint with LiveView socket at `/live`
2. Static file serving from `priv/static`
3. Configured to use Bandit adapter (via config)

### 4. Router (`lib/stock_plan_web/router.ex`)

**Responsibility**: Route definitions. M1 defines only the root route.

```elixir
defmodule StockPlanWeb.Router do
  use StockPlanWeb, :router

  pipeline :browser do
    plug :accepts, ["html"]
    plug :fetch_session
    plug :fetch_live_flash
    plug :put_root_layout, html: {StockPlanWeb.Layouts, :root}
    plug :protect_from_forgery
    plug :put_secure_browser_headers
  end

  scope "/", StockPlanWeb do
    pipe_through :browser

    live "/", HomeLive
  end
end
```

**Behavior**:
1. Single browser pipeline with standard Phoenix plugs
2. Root route `/` renders `HomeLive` (LiveView)
3. Future modules (M9–M12) will add routes: `/upload`, `/portfolio`, `/income`, `/timeline`

### 5. Home LiveView (`lib/stock_plan_web/live/home_live.ex`)

**Responsibility**: Landing page LiveView — validates that LiveView + Tailwind are working.

```elixir
defmodule StockPlanWeb.HomeLive do
  use StockPlanWeb, :live_view

  @impl true
  def mount(_params, _session, socket) do
    {:ok, socket}
  end

  @impl true
  def render(assigns) do
    ~H"""
    <div class="max-w-4xl mx-auto py-12 px-4">
      <h1 class="text-3xl font-bold text-gray-900">Stock Plan Manager</h1>
      <p class="mt-4 text-gray-600">
        RSU, ESPP, and Stock Option management. Upload your E*Trade Benefit History to get started.
      </p>
    </div>
    """
  end
end
```

**Behavior**:
1. Renders app name and brief description
2. Uses Tailwind v4 classes to validate CSS pipeline
3. No database interaction — pure rendering
4. Will be replaced or expanded in M9 (Upload UI)

## Configuration

### `config/config.exs`

```elixir
# Common config
config :stock_plan,
  ecto_repos: [StockPlan.Repo]

config :stock_plan, StockPlanWeb.Endpoint,
  render_errors: [
    formats: [html: StockPlanWeb.ErrorHTML],
    layout: false
  ]
```

### `config/dev.exs`

Only override what the generator doesn't provide. Phoenix 1.8 generates Bandit by default — do NOT add a redundant `adapter:` line if it's already present. The only required override is the port number and DB path.

```elixir
# Port override (generator defaults to 4000)
config :stock_plan, StockPlanWeb.Endpoint,
  http: [ip: {127, 0, 0, 1}, port: 4002]
  # NOTE: Do NOT add `adapter: Bandit.PhoenixAdapter` if already generated.
  # Only add it if the generated config uses a different adapter.

# SQLite at tmp/
config :stock_plan, StockPlan.Repo,
  database: Path.expand("../tmp/stock_plan_dev.db", __DIR__)
```

### `config/test.exs`

```elixir
config :stock_plan, StockPlan.Repo,
  database: Path.expand("../tmp/stock_plan_test.db", __DIR__),
  pool: Ecto.Adapters.SQL.Sandbox

config :stock_plan, StockPlanWeb.Endpoint,
  http: [ip: {127, 0, 0, 1}, port: 4002],
  # NOTE: Do NOT add `adapter:` if already generated with Bandit.
  server: false
```

## Correctness Properties

### Property 1: Server Boot

*For any* fresh clone of the repository, running `mix deps.get && mix ecto.create && mix ecto.migrate && mix phx.server` should result in a server listening on port 4002 that responds to HTTP requests.

**Validates:** Requirement 2, Criteria 2-3; Requirement 7, Criteria 1-6

### Property 2: Zero Application Warnings

*For any* clean compile (`mix compile`), the compiler should produce zero warnings from application code. Warnings originating from external dependencies are excluded but should be documented.

**Validates:** Requirement 1, Criteria 6

### Property 3: Root Route Reachable

*For any* running Dev_Server, `GET http://localhost:4002/` should return HTTP 200 with HTML containing "Stock Plan Manager".

**Validates:** Requirement 5, Criteria 1, 3

### Property 4: Database Lifecycle

*For any* fresh environment, the sequence `mix ecto.create` -> `mix ecto.migrate` -> `mix ecto.reset` should complete without errors, and the SQLite file should exist at the configured path after `ecto.create`.

**Validates:** Requirement 3, Criteria 2-4; Requirement 7, Criteria 3-5

## Error Handling

| Error Condition | Handling Strategy | Impact |
|---|---|---|
| Port 4002 already in use | Phoenix raises on boot; developer must free the port | Dev workflow blocked |
| SQLite file permission error | Ecto.create fails; developer must check `tmp/` permissions | DB creation blocked |
| Missing Hex/rebar | `mix deps.get` fails; developer must install Hex | All commands blocked |

No application-level error handling is needed for M1 — these are all developer-environment issues.

## Testing Strategy

| Test Type | What It Covers | Key Scenarios |
|---|---|---|
| Generated tests | Phoenix default generated tests pass | ConnTest for root route |
| Manual verification | Server boots, page renders, CSS works | `mix phx.server` + browser |

M1 testing is primarily "does it boot and render." Automated tests become meaningful starting with M2 (data layer).

## Implementation Notes

- **Convention-first**: Phoenix 1.8 generates Bandit and Tailwind v4 by default. Use generated defaults first — only override if runtime check proves the generated config is wrong. Do not hardcode `adapter: Bandit.PhoenixAdapter` if the generator already wires it.
- The app name `stock_plan` (underscore) maps to module name `StockPlan` (PascalCase). Phoenix handles this automatically.
- SQLite database path uses `Path.expand("../tmp/...", __DIR__)` in config to ensure it resolves relative to the project root regardless of working directory.
- The `tmp/` directory should already be in `.gitignore` from Phoenix generation. Verify.
- **Root route replacement**: Phoenix may generate verified routes and components tied to its starter page. Replace the root route while preserving the generated layout/component structure (`core_components.ex`, `layouts/`). Do not delete generated layout infrastructure.
